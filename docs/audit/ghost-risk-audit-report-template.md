**Gate state:** Phase 2 blocked until deployer wallet funding is confirmed  

---

## 1. Executive Summary

This report evaluates ghost-risk exposure in tokenised real-world asset infrastructure by comparing three layers of truth:

1. **Observed state** — public property and proof data collected by the prover.
2. **Attested decision** — SafeKrypte-signed relay action generated by the submitter.
3. **Executed result** — on-chain CircuitBreaker transaction and receipt generated by the broadcaster.

A ghost-risk event exists when one or more of these layers diverge.

```text
Ghost risk = observed state ≠ attested decision ≠ executed result
```

The current system can produce the first two layers before Amoy deployment. The third layer becomes available only after the deployer wallet is funded and the contract is deployed.

## 4. System Truth Model

### 4.1 Observed State

The observed state is the prover's representation of external reality. It includes asset IDs, expected hashes, actual hashes, status values, and any fetch errors.

Accepted statuses:

| Status | Meaning | Expected relay action |
|---|---|---|
| `fresh` | Actual proof equals expected proof | `updateProof(assetId, deedHash)` |
| `mismatch` | Actual proof differs from expected proof | `tripCircuit(reason)` |
| `unreachable` | Proof endpoint cannot be reached | `tripCircuit(reason)` |

### 4.2 Attested Decision

The attested decision is the submitter's signed representation of what should happen next. It must include:

- action kind;
- asset identifier;
- proof hash or trip reason;
- state run identifier where available;
- timestamp;
- digest;
- SafeKrypte signature;
- key identifier.

### 4.3 Executed Result

The executed result is the on-chain transaction outcome. It must include:

- transaction hash;
- block number;
- gas used;
- attestation digest;
- action replay data;
- key ID used in the attestation.

## 5. Ghost-Risk Taxonomy

| Code | Risk class | Definition | Detection method | Severity |
|---|---|---|---|---|
| GR-01 | Missing proof | Asset has no reachable proof endpoint | Fetcher status `unreachable` | High |
| GR-02 | Hash mismatch | Expected and actual proof hashes differ | Fetcher status `mismatch` | Critical |
| GR-03 | Stale proof | Proof exists but timestamp exceeds freshness window | Metadata timestamp comparison | Medium / High |
| GR-04 | Unsigned decision | Planned action exists without SafeKrypte signature | Attestation file validation | Critical |
| GR-05 | Signature drift | Attestation digest cannot be reconstructed from payload | Digest recomputation | Critical |
| GR-06 | Execution drift | On-chain action does not match attested action | Receipt/action comparison | Critical |
| GR-07 | Duplicate execution | Same digest executed more than once | Receipt store / contract state | High |
| GR-08 | Oracle mismatch | Broadcasting key is not contract oracle | `contract.oracle()` comparison | Critical |
| GR-09 | Circuit non-response | Trip condition exists but circuit remains open | Post-broadcast state check | Critical |
| GR-10 | Custody gap | Transaction signer remains hot-wallet based | Environment / key custody review | Medium pre-PMF; High production |

## 6. Audit Procedure

### Step 1 — Generate observed state

```bash
node prover/fetcher.js
```

Expected artifact:

```text
.local/state/prover-state.json
```

Validation checks:

- file exists;
- JSON parses;
- `results` is an array;
- each result has `assetId` and `status`;
- status is one of `fresh`, `mismatch`, or `unreachable`.

---

### Step 2 — Generate signed attestations

```bash
node prover/submitter.js --dry-run
```

Expected artifact:

```text
.local/state/submitter-attestations.json
```

Validation checks:

- file exists;
- JSON parses;
- contains `{ attestations: [...] }`;
- each attestation has `digest`, `signature`, `keyId`, and `action`;
- action kind is valid;
- update proof actions contain valid `bytes32` asset ID and deed hash;
- trip actions contain a non-empty reason.

---

### Step 3 — Dry-run broadcaster

```bash
node prover/broadcaster.js --dry-run
```

Validation checks:

- loads attestations;
- validates action schema;
- validates environment requirements;
- verifies oracle identity when connected to deployed contract;
- sends no transaction in dry-run mode.

---

### Step 4 — Live broadcast after Phase 2 gate opens

```bash
node prover/broadcaster.js
```

Expected artifact:

```text
.local/state/broadcast-receipts.json
```

Validation checks:

- each successful tx maps to one attestation digest;
- skipped entries explain idempotency reason;
- failed entries preserve error reason;
- update proof transactions match `latestProof(assetId)`;
- trip transactions change circuit state as expected.
